# Contract — `ProjectGraph` & `ProjectGraphBuilder`

**Status: implemented (Phase 1).** Source of truth: `src-tauri/src/contract/`.
The TS mirror in `src/domain/graph/` is generated by `ts-rs` (`npm run check` regenerates it).

## Responsibility

The single pure-data value that crosses the Tauri IPC boundary. No behavior, no
Tauri/React imports. Everything upstream (Rust analysis) and downstream (React
render/layout) depends only on this shape, so the two sides evolve independently
and are tested against one golden JSON.

Types are defined in [TECHNICAL-DESIGN §3](../plans/TECHNICAL-DESIGN.md). Entities:
`GroupNode`, `ModuleNode`, `Edge`, `Diagnostic`, `Annotation`, `ProjectGraph`.

## `ProjectGraphBuilder` — invariant gate

`builder.rs` accumulates parts; `build()` returns `Result<ProjectGraph, BuildError>`.
Validation (`validate.rs`) enforces the five §2.2 invariants. **Construct a graph
only through the builder** so no unchecked `ProjectGraph` reaches the contract.

| `BuildError` | Invariant | Concrete check |
|--------------|-----------|----------------|
| `SiblingOverlap` | 1 — containment is a tree | every `parentId` exists; no parent cycle |
| `MultiGroupMembership` | 2 — module in exactly one group | `groupId` (when set) references an existing group |
| `ForeignFacade` | 3 — facade belongs to its group | each `facadeModuleIds` entry exists and its `groupId` is that group |
| `DanglingEdge` | 4 — edge endpoints exist | `source`/`target` reference existing module ids |
| `NondeterministicId` | 5 — stable, unique ids | no empty ids; no duplicate ids across groups/modules/edges |

The data model already makes invariants 1 and 2 *partly* structural: a module
carries a single `groupId` and a group a single `parentId`, so a module cannot
literally live in two groups and containment cannot be a DAG. The builder enforces
the residual teeth (referential existence + acyclicity). See
[lessons-learned](../lessons-learned/single-parent-collapses-overlap-invariants.md).

## Determinism

Ids are path-derived; edge id = `${source}->${target}:${kind}:${ordinal}`. No
timestamps/random in graph data, so golden-fixture diffs and layout snapshots are
meaningful. Analyzers (Phase 4+) must emit in a deterministic order.

## North Star fixtures

- `tests/fixtures/ts-basic-project/` — the reference TS project (the analysis input).
- `tests/fixtures/golden/project-graph.json` — its hand-authored expected output.
  Both Rust (`contract::contract_tests`) and TS (`tests/contract.test.ts`) suites
  parse it; the Rust suite also re-feeds it through the builder to prove it is valid.
  Scope/decisions for the golden file are documented in the fixture's `README.md`.
